💡 Architectural Benchmarking Insights

PostgreSQL-backed distributed caching achieves excellent performance when structured correctly. The benchmarks above demonstrate critical database and cache design concepts:

1. UNLOGGED Tables vs. Write-Ahead Logging (WAL)

By default, `cache-postgres` sets use_wal=False, creating an UNLOGGED PostgreSQL table. This disables disk transaction logs, increasing write speeds immensely (getting closer to Redis/Valkey). Using WAL logging secures data durability but incurs heavy disk I/O penalties on updates/inserts.

2. Advisory Lock Stampede Protection

Under a "Cache Stampede", massive concurrent clients fetch the same expired key, crashing database backends with duplicated calculations. `cache-postgres` resolves this in get_or_create by acquiring an exclusive pg_advisory_lock. Concurrent calls wait and load the value computed by worker #1 instantly.

3. Connection Pool Sizing Impact

Caching operations are short and highly concurrent. A single-connection constraint (Pool=1) leads to high query contention and queuing, skyrocketing P95/P99 latency. Distributing query tasks over a reasonably sized Connection Pool (Pool=10) yields massive parallel throughput gains.

4. Valkey (In-Memory) vs. Postgres (Relational)

Valkey, being written in highly optimized C and operating entirely in memory, serves as a performance baseline. With optimized pooling and unlogged tables, `cache-postgres` brings relational-based caching surprisingly close to in-memory performance while adding native SQL query capabilities.
